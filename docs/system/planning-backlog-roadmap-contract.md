# Planning Bullets + Backlog + Roadmap Contract

## Purpose

Define the canonical naming, authority boundaries, storage locations, and linking expectations for the
primary Planning workflow centered on repo-backed Planning Bullets, Repository Backlog docs, Roadmap
docs, and Plan Pack handoff while keeping typed intake, external Obsidian notes, and legacy
planning-record artifacts in bounded compatibility roles.

## Canonical Names

- **Repository Backlog** — the canonical repo-backed backlog artifact family for planning intake and queued work. Primary per-session artifacts live under `docs/backlogs/*.md`; `docs/backlog.md` remains a legacy compatibility surface.
- **Roadmap** — the canonical phased planning document that sits above plan packs.
- **Planning Bullets** — the canonical repo-backed markdown seed file for future-session freeform bullets.
- **Roadmap Sync** — the canonical name for automatic deterministic reconciliation between linked
  roadmap items, backlog items, and plan packs.
- **Plan Pack** — the existing execution artifact generated by `@o-planner`; this name and role stay
  unchanged.
- **Typed Planning Intake** — a supported compatibility surface for structured requests that is no longer part of the primary Planning-tab authoring workflow and should live under an explicit Planning compatibility/operator area when surfaced in UI.
- **External Obsidian notes** — a supported external/non-canonical compatibility surface, not a primary Planning-tab surface, and when surfaced in UI they belong in the Planning compatibility/operator area.
- **Legacy planning-record notes** — compatibility-only historical planning persistence, not a primary Planning-tab surface, and when surfaced in UI they belong in the Planning compatibility/operator area.

Do not introduce alternate product names for these same concepts in canonical docs.

## Authority Boundaries

| Artifact | Canonical authority | Primary location | Scope | Not authoritative for |
| --- | --- | --- | --- | --- |
| Planning Bullets | Repo docs file | `<repo>/docs/planning/bullets.md` | Freeform future-plan seeds below backlog acceptance and direct seed input for roadmap/plan handoff | Accepted backlog state, active execution |
| Repository Backlog | Repo docs artifact family | `<repo>/docs/backlogs/<session-slug>.md` (primary), `<repo>/docs/backlog.md` (legacy compatibility) | Repo-wide intake, triage, queued work, and session-close carryover | Session execution state, work-unit decomposition |
| Roadmap | Repo docs files | `<repo>/docs/roadmaps/<roadmap-slug>.md` | Phased outcomes, sequencing, and status across work | Detailed execution instructions |
| Plan Pack | Session-state artifact | `~/.copilot/session-state/<SESSION_ID>/plan.md` | One execution session | Repo backlog ownership or roadmap prioritization |
| Typed Planning Intake | Repo docs files | `<repo>/docs/planning/intake/*.json` | Structured request compatibility surface for audit, roadmap-request, review-prep, commit-prep, and related workflows | Primary Planning-tab authoring workflow, accepted backlog state |
| External Obsidian notes + mirrors | External/non-canonical compatibility surface | Local Obsidian vault configured for `copilot-ui` | Supplemental notes plus deterministic mirror representations of canonical bullets/roadmaps for the selected repo | Canonical backlog, roadmap, bullets, or active execution authority |
| Legacy planning-record notes | Local planning persistence | `copilot-ui` planning DB plus record artifact routes | Historical/compatibility-only planning notes | Primary Planning-tab authoring workflow or new backlog/roadmap authority |

Repo context for Repository Backlog and Roadmap resolution MUST reuse the existing Catalog repo
selection/source flow in `copilot-ui`; Planning must not create a separate repo registry or competing
repo-selection authority.

External Obsidian notes may be surfaced inside Planning for the selected Catalog repo, but only in an
explicit compatibility/operator area and they MUST be labeled as external/non-canonical everywhere they
are shown. Reading or seeding from that surface does not change canonical authority: repo docs and the
active session `plan.md` still win.

Explicit operator-driven promotion from an external Obsidian note into canonical backlog or roadmap docs
is allowed, but once that promotion happens the canonical authority lives in the repo docs and does not
flow back from the note.

SAFE interpretation for canonical planning mirrors:

- Obsidian may host deterministic mirror notes for `docs/planning/bullets.md` and `docs/roadmaps/*.md`.
- Those mirror notes are a convenience representation only, not an alternate write-authority lane.
- Mirror generation must be repo-scoped, deterministic, and refreshable from canonical repo artifacts.
- Mirror metadata or freshness checks may be read, but mirror bodies must not be parsed back into
  canonical authority.

## Orchestrator Planning Model

The orchestrator planning model is intentionally layered around four primary planning surfaces:

1. **Planning Bullets** capture candidate ideas and pre-backlog seeds.
2. **Repository Backlog** is the durable repo-wide queue and triage surface.
3. **Roadmap** organizes selected backlog work into phased outcomes and sequencing.
4. **Plan Pack** turns one approved slice into active execution work for a single session.
5. **Follow-up / carryover issue docs** capture post-run unresolved goals, planning ideas, and
   out-of-scope findings that should persist outside the active session.

Primary supported Planning-tab workflow:

1. start with `docs/planning/bullets.md`
2. promote selected work into `docs/backlogs/*.md` or reference existing `RB-*` items
3. organize selected backlog work into `docs/roadmaps/*.md`
4. hand one approved slice into `plan.md`

Plan Packs may also be seeded directly from bullets, backlog items, or roadmap items when explicit
`PB-*`, `RB-*`, and `RM-*` IDs are preserved.

Bullet cleanup remains manual-only by default. When bullets feed backlog, roadmap, or plan creation,
the supported workflow is to preserve the bullet and update its explicit linked references unless the
user explicitly asks for scoped deletion.

Typed intake, external Obsidian notes, and legacy planning-record notes may still exist as
compatibility inputs, but they are non-primary surfaces for the Planning tab and must not displace the
four-surface workflow above. When UI keeps them available, it should do so behind an explicit
compatibility/operator area rather than mixing them into the main workflow tabs.

This means the orchestrator's active execution TODO state is not the same authority as the durable repo
backlog:

- active execution TODO state lives in the current plan pack, progress tracker, chat-first session
  state, or equivalent persisted session artifacts
- durable repo backlog state lives in backlog/roadmap/issue-doc surfaces under the selected repo

The visible task board used by `copilot-ui` is a separate execution-control projection over durable
repo-state tasks, not a replacement for the Repository Backlog:

- task-board authority lives under `~/.copilot/repo-state/<repoId>/tasks/` and `tasks.archive/`
- the board may drive active/queued execution and auto-triggered local workflow behavior
- bounded ephemeral UI state may exist outside canonical task storage, but the board must not become a
  second durable backlog or roadmap authority
- workflow automation launched from the board remains local-only for MVP; packaged n8n is the favored
  direction, but the exact delivery shape remains validation-dependent

The orchestrator may read repo planning surfaces to frame or close work, but it must not pretend that a
session plan pack replaces repo backlog ownership or roadmap prioritization.

## Planning Surface Selection Contract

The orchestrator and canonical docs use the same normalized route-selection fields:

| Field | Allowed values | Contract |
| --- | --- | --- |
| `planning_surface` | `plan-pack` \| `roadmap` \| `both` \| `none` | Selects the planning surface, if any, for the request |
| `session_horizon` | `single-session` \| `multi-session` | Declares whether the work is expected to close in one execution session or span durable follow-up |
| `execution_readiness` | `ready` \| `stageable` \| `not-ready` | Declares whether execution can start now, must be staged first, or is blocked on more input |
| `overlap_risk` | `low` \| `medium` \| `high` | Signals the risk of mixing durable planning with active execution or creating bounded validation overlap |

Interpretation rules:

- `planning_surface: roadmap` is the durable multi-session planning surface. It is authoritative for phased outcomes and sequencing across sessions.
- `planning_surface: plan-pack` is the session-scoped execution surface. It is authoritative only for the active execution slice.
- `planning_surface: both` means durable roadmap work happens first, then a plan pack is generated for the active session using the linked durable IDs (`PB-*`, `RB-*`, and `RM-*` when present).
- `planning_surface: none` means no roadmap or plan-pack artifact is required for the request.

These route selections do not create hidden memory or change roadmap authority. Roadmap remains the durable repo-planning surface; plan packs remain session execution artifacts.

Delivery-oriented requests such as commit prep, review prep, and CI result checks are valid request classes in this contract. They often use `planning_surface: none`, unless the user explicitly asks for durable roadmap work or active execution planning. They do not imply push automation or remote CI writes.

### Explicit Orchestrator Outcomes

The orchestrator MUST treat the four planning-surface outcomes as explicit, first-class routing results:

| Outcome | Required route selection | Scope | Authority note |
| --- | --- | --- | --- |
| Roadmap-only | `planning_surface: roadmap` | Durable phased planning that spans multiple sessions | Roadmap stays above plan packs and remains authoritative for sequencing across sessions |
| Plan-pack-only | `planning_surface: plan-pack` | One active execution slice | `plan.md` is authoritative only for the current session execution slice |
| Both | `planning_surface: both` | Durable roadmap/backlog framing followed by one active execution slice | Roadmap remains the durable authority; the generated plan pack inherits linked IDs for the selected slice |
| None | `planning_surface: none` | Requests that do not require roadmap or plan-pack artifacts | Use no planning artifact by default; common for bounded delivery/reporting requests |

Default interpretation notes:

- roadmap-only implies `session_horizon: multi-session` unless the caller explicitly constrains it otherwise
- plan-pack-only usually implies `session_horizon: single-session`, though it may remain `stageable` before execution starts
- both explicitly separates durable multi-session roadmap authority from session-local execution authority
- none is valid for commit prep, review prep, CI result checks, and similar delivery/reporting work that should not create backlog or execution artifacts unless explicitly requested

## File Locations

For a selected repository root `<repo>/`:

- **Planning Bullets** live at `<repo>/docs/planning/bullets.md`.
- **Typed Planning Intake** lives under `<repo>/docs/planning/intake/`.
- **Repository Backlog** primarily lives under `<repo>/docs/backlogs/`.
- `docs/backlogs/<session-slug>.md` is the primary per-session Repository Backlog artifact for session-close carryover.
- `docs/backlog.md` remains supported as a legacy compatibility artifact for agents, validators, and older docs flows.
- **Roadmaps** live under `<repo>/docs/roadmaps/`.
- Multiple roadmap files are supported.
- Backlog filenames SHOULD use lowercase kebab-case session slugs:
  - `<repo>/docs/backlogs/2026-04-03-session-close.md`
  - `<repo>/docs/backlogs/platform-audit-follow-up.md`
- Roadmap filenames SHOULD use lowercase kebab-case slugs:
  - `<repo>/docs/roadmaps/platform-foundation.md`
  - `<repo>/docs/roadmaps/q2-delivery.md`

Session-close Repository Backlog artifacts should persist durable carryover under clear categories that
cover:

- Work Not Done
- Issues
- Suggestions

Entries in those categories may introduce new `RB-*` items or explicitly reference existing `RB-*`
items. Do not invent a new backlog ID family for per-session backlog files.

Plan packs remain session-state artifacts, not repo docs:

- `~/.copilot/session-state/<SESSION_ID>/plan.md`

This contract does not replace the existing session-state plan-pack contract documented in
[docs/system/session-state-artifacts.md](docs/system/session-state-artifacts.md) and
[engine-assets/skills/planpack-authoring/SKILL.md](engine-assets/skills/planpack-authoring/SKILL.md).

## Stable ID + Linking Expectations

Stable linked IDs are required for automatic reconciliation.

- **Bullet IDs** are canonical machine identifiers with format `PB-###`.
  - Example: `PB-001`
  - Unique within the repository bullets file
  - Stable after creation
- **Backlog item IDs** are canonical machine identifiers with format `RB-###`.
  - Example: `RB-001`
  - Unique across the Repository Backlog artifact family (`docs/backlogs/*.md` plus legacy `docs/backlog.md` when present)
  - Stable after creation
- **Roadmap item IDs** are canonical machine identifiers with format `RM-<roadmap-slug>-###`.
  - Example: `RM-platform-foundation-001`
  - `<roadmap-slug>` matches the roadmap filename slug
  - Stable after creation
- **Plan-pack IDs** keep the existing contracts:
  - session ID: `YYYYMMDD_HHMMSS_RAND4`
  - group ID: `G-##-slug`
  - work unit ID: `WU-###`

Linking rules:

1. A roadmap item MUST list the backlog item IDs it covers; prose-only association is not sufficient.
2. When roadmap creation or direct plan handoff starts from bullets, preserve the originating `PB-*` IDs explicitly in roadmap notes or plan markers instead of collapsing them into prose.
3. When the artifact model supports bullet-side linked references, bullet-driven backlog, roadmap, and plan creation SHOULD update those explicit `RB-*`, `RM-*`, and plan references on the originating bullet.
4. A plan pack created from bullets, backlog, or roadmap work MUST preserve the linked `PB-*`, `RB-*`,
   and `RM-*` IDs verbatim in explicit plan references when those origins exist.
5. Roadmap Sync MUST key reconciliation off explicit IDs and repo-relative file locations, not heading
   text, timestamps, or semantic text matching.
6. If a plan pack has no linked backlog/roadmap IDs, automatic reconciliation MUST fail closed rather
   than guess.

### Canonical Plan-Pack Link Markers

When a Plan Pack is intended to participate in Roadmap Sync, it MUST preserve linkage with explicit
markdown markers in `plan.md`. The canonical marker names are:

- `IE_LINKED_BACKLOG_IDS`
- `IE_LINKED_ROADMAP_IDS`
- `IE_LINKED_BULLET_IDS`
- `IE_PLAN_REF` or `IE_SESSION_REF`
- `IE_PLAN_ORIGIN_KIND`
- `IE_PLAN_OUTCOME`

Contract rules:

1. `IE_LINKED_BACKLOG_IDS` lists canonical `RB-*` IDs.
2. `IE_LINKED_ROADMAP_IDS` lists canonical `RM-*` IDs.
3. `IE_LINKED_BULLET_IDS` lists canonical `PB-*` IDs when a plan was seeded from repo bullets.
4. `IE_PLAN_REF` is the preferred explicit plan reference marker; `IE_SESSION_REF` remains an accepted
   compatibility alias.
5. `IE_PLAN_ORIGIN_KIND` uses `bullet`, `backlog`, `roadmap`, `synced-note`, or `direct`.
6. `IE_PLAN_OUTCOME` uses `completed`, `superseded`, or `abandoned`.
7. If these markers are absent, malformed, or inconsistent with roadmap coverage, Roadmap Sync MUST fail
   closed.

## Backlog vs Roadmap vs Plan Pack Responsibilities

| Artifact | Primary responsibility | Questions it answers | Must not replace |
| --- | --- | --- | --- |
| Planning Bullets | Capture low-friction future-session ideas before they are accepted | What should we remember and seed into future plans or backlog work? | Accepted backlog state |
| Repository Backlog | Capture candidate work and repo-scoped planning intake | What work exists? What still needs triage or scheduling? | Roadmap phasing or execution specs |
| Roadmap | Organize selected work into phased outcomes | What outcomes matter next? How does backlog work roll up into phases? | Step-by-step implementation planning |
| Plan Pack | Decompose one selected slice into executable work units | How should this work be executed now? In what order? With what validation? | Repo backlog or roadmap portfolio state |
| Typed Planning Intake | Capture structured compatibility requests | What audit, roadmap-request, review-prep, commit-prep, or CI result check work has been requested outside the primary Planning-tab flow? | Freeform bullet capture, accepted backlog state, or primary Planning-tab workflow |

Operationally:

- Bullets are the browse-first pre-plan seed layer.
- Backlog is the canonical repo TODO/backlog surface.
- Roadmap is the canonical phase/outcome surface above execution.
- Plan Pack remains the canonical execution artifact.
- Typed intake, external Obsidian notes, and legacy planning-record notes are compatibility-only planning inputs. They may stay implemented for interoperability, but they are not primary Planning-tab surfaces and should be isolated behind an explicit Planning compatibility/operator area in UI.

The orchestrator should therefore move from planning bullets/intake toward backlog and roadmap for
durable planning, then into a plan pack for active execution, and finally back out to follow-up or
carryover docs only when the post-run work is no longer active session execution.

## Logical Planning Surfaces and Default Paths

Logical planning surfaces are the conceptual planning/carryover slots the orchestrator may target.
Today, this contract defines the default repo/session paths listed below plus the implemented External
Obsidian Notes compatibility surface. Other provider-specific routing or storage locations remain future
extensions and must not be implied unless separately implemented and approved.

| Logical surface | Today’s default path | Primary authority | Notes |
| --- | --- | --- | --- |
| Planning Bullets | `<repo>/docs/planning/bullets.md` | Repo docs | Freeform seed ideas before backlog acceptance |
| Repository Backlog | `<repo>/docs/backlogs/*.md` (primary), `<repo>/docs/backlog.md` (legacy compatibility) | Repo docs | Durable queued/triaged repo work plus per-session carryover |
| Roadmaps | `<repo>/docs/roadmaps/*.md` | Repo docs | Phased outcomes above execution |
| Active Plan Pack | `~/.copilot/session-state/<SESSION_ID>/plan.md` | Session artifact | Active execution plan, not durable repo backlog |
| Typed Planning Intake | `<repo>/docs/planning/intake/` | Repo docs | Compatibility-only typed structured requests exposed from the Planning compatibility/operator area when needed |
| External Obsidian Notes | Configured Obsidian vault path for selected repo context | External/non-canonical | Compatibility-only note-reading surface plus deterministic mirrors of canonical bullets/roadmaps, exposed from the Planning compatibility/operator area when needed |
| Unresolved Goals | `<repo>/docs/issues/unresolved-goals.md` | Repo docs | Non-active unresolved goal carryover |
| Planning Ideas Log | `<repo>/docs/issues/planning-ideas-log.md` | Repo docs | Cross-session planning ideas not accepted as active work |
| Out-of-Scope Findings | `<repo>/docs/issues/out-of-scope-findings.md` | Repo docs | Discoveries intentionally outside current approved scope |

Fail-closed rules:

1. If the selected repo root is missing or unsupported, do not guess an alternate planning location.
2. If a required logical surface path is absent, treat the surface as unavailable rather than inventing
   a substitute path.
3. If linkage between plan packs and repo planning docs is missing or malformed, reconciliation MUST
   fail closed.
4. If deterministic Obsidian mirror metadata is malformed or a mirror cannot be written safely, mirror
   refresh MUST fail closed instead of silently overwriting or inventing fallback content.
5. Future provider-specific routing may define alternate physical locations, but no such routing should
   be implied in current prompts or docs unless separately implemented and approved.

## Roadmap Sync Contract

Roadmap Sync is an automatic deterministic reconciliation flow, not a manual note-maintenance habit.

High-level contract:

1. **Trigger**
   - Runs when a linked plan pack reaches a terminal execution outcome such as completed, superseded,
     or abandoned.
2. **Inputs**
   - active repo from Catalog repo context
   - linked roadmap item IDs
   - linked backlog item IDs
   - terminal plan outcome plus plan reference
   - Plan Pack link markers recorded in the session `plan.md`
3. **Effects**
   - update the linked roadmap item state
   - record which plan satisfied, superseded, or abandoned that roadmap item
   - update the linked backlog item state to reflect satisfaction or follow-up need
4. **Safety**
   - deterministic and idempotent
   - no heuristic text matching
   - no silent cross-repo mutation
   - missing IDs or broken links fail closed for manual repair

Default reconciliation stance:

- successful completion should mark the roadmap item satisfied/done and mark the linked backlog item
  satisfied rather than rely on manual cleanup
- superseded or abandoned execution should preserve traceability and avoid pretending the work was done
- physical deletion of satisfied backlog items is optional follow-up compaction, not the default sync
  behavior
- physical deletion of source bullets is not default sync behavior; prefer retaining the bullet and recording linked backlog or plan references

Current dashboard/API entrypoint:

- `POST /api/sessions/:id/roadmap-sync`
  - reads the selected session `plan.md`
  - resolves the repo via Catalog repo context
  - applies deterministic roadmap/backlog reconciliation when the linked plan has reached a terminal
    outcome

## Legacy Coexistence + Migration

Legacy planning-record research notes and diagrams may continue to exist for older records through the
current planning persistence APIs, but they are no longer the canonical planning surface for the new
workflow. When still exposed in UI, they belong in the Planning compatibility/operator area.

These record-scoped research/diagram fields and their legacy alias properties are deprecated
compatibility surfaces. They remain readable for older records, but must not be treated as canonical
inputs for new backlog, roadmap, or plan-pack flows.

Migration stance:

- no dual-write requirement between repo docs and planning-record notes
- new bullet/intake/backlog/roadmap workflows should write repo docs only
- useful legacy notes may be promoted into `docs/backlogs/<session-slug>.md`, legacy `docs/backlog.md` when compatibility is required, or roadmap files
- until promoted, legacy notes remain historical context and compatibility data only
- legacy record-scoped research artifacts must not override repo-backed backlog or roadmap docs
