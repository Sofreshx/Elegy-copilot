# Spec-Driven Development Contract

## Intent

Define the authoritative, verifiable contract for durable repo specs under `docs/specs/`. Every spec in this system — including this one — MUST conform to this contract. This spec is the normative source of truth; skills, docs, and validators derive their requirements from it.

## Context Evidence

- `catalog-assets/shared-skills/spec-authoring/SKILL.md` (190 lines) — inline "shared contract" defining required frontmatter, headings, and accepted values. This skill will reference this spec instead of duplicating the contract.
- `catalog-assets/shared-skills/spec-review/SKILL.md` (52 lines) — 16 review checks; check #1 says "Verify the spec uses the shared contract and required headings" but the contract is not defined in a spec artifact.

- `catalog-assets/shared-skills/spec-planning-bridge/SKILL.md` (99 lines) — defines the file-scope selector grammar for elegy-planning handoff inline rather than referencing a spec.
- `scripts/validate-specs.js` (878 lines) — enforces the contract structurally; its checks define what the contract IS in practice. The validator should reference this spec as its compliance target.
- `scripts/validate-specs.test.js` (508 lines) — 16 test cases; validates against the current validator behavior, not against a spec.

## Requirements

### Allowed Behavior

#### R1 — Durable Path

- R1.1: Every durable spec MUST live at `docs/specs/<spec-slug>/spec.md` where `<spec-slug>` matches the `spec_id` frontmatter key.
- R1.2: An optional catalog MAY exist at `docs/specs/index.md`, generated by `scripts/generate-spec-index.js`.
- R1.3: Per-type templates MAY exist at `docs/specs/_templates/<type>.md`.

#### R2 — Required Frontmatter

- R2.1: Every spec MUST include these 5 frontmatter keys: `spec_id`, `title`, `status`, `type`, `updated`.
- R2.2: `spec_id` MUST be a kebab-case slug unique across all specs in the repo.
- R2.3: `title` MUST be a non-empty string.
- R2.4: `status` MUST be one of: `draft`, `approved`, `implemented`, `superseded`, `abandoned`.
- R2.5: `type` MUST be one of: `feature`, `workflow`, `contract`, `skill`, `agent`, `migration`.
- R2.6: `updated` MUST be a valid ISO-8601 date in YYYY-MM-DD format.
- R2.7: Optional frontmatter keys include: `supersedes` (list of `spec_id` strings), `superseded_by` (single `spec_id` string), `freshness: ignore` (skip staleness warnings), `liveness_skip_paths` (list of path patterns to skip in liveness checks), `created`, `approved_at`, `implemented_at`, `superseded_at`, `abandoned_at` (all ISO-8601 dates).

#### R3 — Required Headings

- R3.1: Every spec MUST include these 8 sections as level-2 headings (e.g., `## Intent`):
  1. `Intent`
  2. `Context Evidence`
  3. `Requirements`
  4. `Non-Goals`
  5. `Acceptance Checks`
  6. `Implementation Links`
  7. `Validation Evidence`
  8. `Drift Notes`
- R3.2: `Intent` MUST contain non-empty content describing the problem or opportunity — not "none", "todo", "wip", "tbd", or similar placeholders.
- R3.3: `Context Evidence` MUST list concrete file paths, commands, docs, tests, or runtime data with reasons justifying the spec.
- R3.4: `Requirements` MUST contain unambiguous, testable requirements.
- R3.5: `Non-Goals` MUST cover likely scope-creep edges.
- R3.6: `Implementation Links` MUST list files, tests, PRs, or plans that materially connect to the spec.
- R3.7: `Validation Evidence` MUST contain real evidence when `status: implemented` — not a placeholder (e.g., "pending", "none yet", "todo").
- R3.8: `Drift Notes` MUST record deviations, follow-up reconciliation, or "None."

#### R4 — Acceptance Checks

- R4.1: Every spec MUST have at least 2 acceptance checks in the `## Acceptance Checks` section.
- R4.2: Each acceptance check bullet MUST be immediately followed by a `→ verify:` line (indented 2 spaces, non-empty content) that specifies a concrete verification method: a test command, script path, or manual steps.
- R4.3: Acceptance checks MUST NOT contain vague language — the system MUST reject tokens matching `AC_VAGUE_TOKEN_RE` (defined in `scripts/lib/ac-vague-tokens.js`).

#### R5 — Allowed/Forbidden Behavior Subsections

- R5.1: The `## Requirements` section MUST contain `### Allowed Behavior` and `### Forbidden Behavior` subsections.
- R5.2: `Allowed Behavior` MUST describe what the system SHOULD do — positive requirements.
- R5.3: `Forbidden Behavior` MUST describe what the system MUST NOT do — boundary conditions, error states, and excluded paths.

#### R6 — Spec Lifecycle

- R6.1: A spec's `status` follows this lifecycle:
  - `draft` → `approved` → `implemented`
  - `draft` → `abandoned` (terminal)
  - `approved` → `superseded` (terminal)
  - `implemented` → `superseded` (terminal)
- R6.2: `draft`: Spec is being authored; content is provisional. Required frontmatter only.
- R6.3: `approved`: Spec has passed review and is ready to anchor planning. `approved_at` recommended. MUST link to an active plan or work point.
- R6.4: `implemented`: Requirements have been met; acceptance checks pass. `Validation Evidence` MUST be non-empty. `implemented_at` recommended. Spec remains as the permanent requirements record.
- R6.5: `superseded`: Spec is replaced by a newer spec. `superseded_by` REQUIRED. `superseded_at` recommended. Terminal status.
- R6.6: `abandoned`: Reviewed decision not to implement the spec. `abandoned_at` recommended. Terminal status. MUST NOT have `superseded_by`.

#### R7 — Cross-Spec Relationships

- R7.1: A spec MAY declare it supersedes another via `supersedes: <spec_id>` (or list of IDs) in frontmatter.
- R7.2: A spec MUST declare `superseded_by: <spec_id>` when `status: superseded`.
- R7.3: A spec MUST NOT set both `supersedes` and `superseded_by` in the same spec.
- R7.4: All `supersedes` and `superseded_by` values MUST reference real `spec_id` values in other spec files in the repo.
- R7.5: Supersedes chains MUST be acyclic (no A → B → A).
- R7.6: When spec A declares `supersedes: B`, spec B MUST have `status: superseded` with `superseded_by: A` (bidirectional validation).
- R7.7: Non-authoritative relationships (related but not superseding) MUST be noted in `Drift Notes` or `Context Evidence` prose.

#### R8 — Plan.md Requirement

- R8.1: A sibling `plan.md` at `docs/specs/<slug>/plan.md` is REQUIRED when the spec has 5+ requirements bullets in `## Requirements` or 2+ implementation phases.
- R8.2: The `plan.md` MUST contain the execution plan: implementation order, risk assessment, and validation steps.
- R8.3: The validator SHALL produce an advisory warning (not error) when a `draft` or `approved` spec has 5+ requirements but no sibling `plan.md`.

#### R9 — Validation Evidence

- R9.1: When `status: implemented`, `Validation Evidence` MUST contain non-empty, concrete evidence — not "pending", "none yet", or "todo".
- R9.2: Evidence SHOULD include test output, build logs, review verdicts, or verified acceptance check results.
- R9.3: When `status` is not `implemented`, `Validation Evidence` MAY be "Pending implementation" or similar placeholder.
- R9.4: The validator MUST reject (error) an `implemented` spec with placeholder Validation Evidence.

#### R10 — File-Scope Selector Contract

- R10.1: The file-scope selector grammar for linking specs to elegy-planning entities is:
  - Format: `<type>:<intent>:<selector>`
  - Type: `exact` (literal path match) or `glob` (pattern match)
  - Intent: `primary` (main artifact), `review`, or `affected`
- R10.2: For spec-to-planning linkage, the canonical selector is: `exact:primary:docs/specs/<spec-slug>/spec.md` where `<spec-slug>` matches the `spec_id` frontmatter key.
- R10.3: This selector is used in: work point `fileScope` arrays, plan `fileScope` arrays, and `Implementation Links` in the spec itself.
- R10.4: Semantic linkage (audit trail) MUST always pair a file-scope selector with a `planning_insight_record` of `insightType: 'spec-link'`.

#### R11 — Spec Retention Rules

- R11.1: Durable specs are the permanent requirements record. Do NOT physically archive or move them to a separate folder.
- R11.2: Review `draft` specs after 90 days of inactivity — promote to `approved`, abandon, or update content.
- R11.3: `approved` specs MUST link to an active plan or work point. Review if unlinked for >180 days.
- R11.4: `implemented` specs are retained permanently. Update `Drift Notes` if behavior changes.
- R11.5: `superseded` specs are retained permanently with `superseded_by`.
- R11.6: `abandoned` specs are retained permanently as a record of a reviewed decision. Must NOT have `superseded_by`.
- R11.7: Delete a spec ONLY when ALL of: (a) accidental duplicate, (b) `status: draft`, (c) no Implementation Links, no Validation Evidence, no planning links. Never delete `approved`, `implemented`, `superseded`, or `abandoned` specs without explicit approval.

#### R12 — Freshness Policy

- R12.1: `draft` specs with `updated` older than 90 days SHALL produce an advisory warning: "stale draft (N days since last update)".
- R12.2: `approved` specs with `updated` older than 180 days SHALL produce an advisory warning: "stale approved spec (N days, review for planning linkage)".
- R12.3: `implemented` specs with `updated` older than 180 days SHALL produce an advisory warning: "stale implemented spec (N days, consider reviewing for drift)".
- R12.4: Freshness warnings are advisory (never cause non-zero exit code).
- R12.5: `abandoned` and `superseded` specs are terminal — no staleness warnings produced.
- R12.6: A spec MAY opt out of freshness warnings by setting `freshness: ignore` in frontmatter.

#### R13 — ADR Promotion

- R13.1: If a spec describes a key architectural decision, workflow-authority boundary, trust model, or long-lived contract change, that decision MUST be promoted to an ADR under `docs/system/<name>-adr.md`.
- R13.2: The ADR file path MUST be referenced in the spec's `Context Evidence`.
- R13.3: The spec-review skill SHALL check for ADR promotion when the spec's scope warrants it.

### Forbidden Behavior

- A spec MUST NOT set both `supersedes` and `superseded_by` in the same spec (R7.3).
- A spec with `status: superseded` MUST NOT omit `superseded_by` (R7.2).
- A spec with `status: abandoned` MUST NOT have `superseded_by` (R6.6).
- A spec MUST NOT have empty or placeholder `Intent` content — no "none", "todo", "wip", "tbd" (R3.2).
- A spec MUST NOT have placeholder `Validation Evidence` when `status: implemented` (R9.1).
- A spec MUST NOT have fewer than 2 acceptance check bullets (R4.1).
- A spec MUST NOT have an acceptance check bullet without a following `→ verify:` line (R4.2).
- A spec MUST NOT use vague tokens in acceptance checks — words matching `AC_VAGUE_TOKEN_RE` (R4.3).
- A spec MUST NOT have circular supersedes chains (R7.5).
- A spec MUST NOT use `spec-as-source` for general product code or broad implementation notes (per spec-dev routing rules).
- A spec MUST NOT be deleted when `status` is `approved`, `implemented`, `superseded`, or `abandoned` without explicit approval (R11.7).

## Non-Goals

- Defining the validator implementation — this spec defines the contract; `scripts/validate-specs.js` enforces it.
- Defining CI, pre-commit hook, or validation pipeline architecture — those are operational concerns, not contract.
- Defining how skills load, route, or invoke each other — that belongs to skills governance.
- Defining elegy-planning database schema or API — that belongs to the elegy-planning contract.
- Defining repo setup or bootstrap procedures — that belongs to repo-setup-governance.
- Defining ADR format or review process — that belongs to adr-governance.
- Defining spec index format — the index is a generated catalog, not part of the spec contract.
- Changing the file-based spec model to a database model.
- Merging specs and elegy-planning — they remain separate working models.
