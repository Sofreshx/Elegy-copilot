# Mobile Local Testing and Integration Audit

This document outlines how to test the Mobile Companion locally, audits the current integration path to the VS Code extension, and summarizes what is required to make local testing and GitHub login reliable.

## Goals
- Validate Mobile Companion behavior locally (UI, API calls, session lists, permissions).
- Verify whether Mobile Companion can connect to the VS Code extension directly or via relay.
- Assess GitHub OAuth login feasibility for local development.

## Observed Architecture (Current)
- Mobile Companion connects to a WebSocket relay using `VITE_RELAY_URL` and sends messages with a wrapper `{ type: "request", payload: { jsonrpc, method, params } }` as seen in [mobile-companion/src/services/relayApi.ts](mobile-companion/src/services/relayApi.ts).
- Mobile OAuth redirects to GitHub, then exchanges the `code` via a relay endpoint `POST /auth/github/callback` as seen in [mobile-companion/src/services/authService.ts](mobile-companion/src/services/authService.ts).
- The VS Code extension exposes a local WebSocket server on `ws://127.0.0.1:<port>` and expects JSON-RPC request objects at the top-level (`{ jsonrpc, id, method, params }`) as seen in [vscode-skill-installer/src/wsServer.ts](vscode-skill-installer/src/wsServer.ts).
- The extension requires a JWT token in the WebSocket request (query param or headers) generated by the extension’s auth manager as seen in [vscode-skill-installer/src/wsAuth.ts](vscode-skill-installer/src/wsAuth.ts).
- The Cloud Relay service currently exposes `/v1/ws` and health endpoints; the OAuth endpoints in the relay API docs are not implemented in code at this time. See [cloud-relay/src/index.ts](cloud-relay/src/index.ts) and [docs/relay-api-reference.md](docs/relay-api-reference.md).

## Local Testing Paths

### Path A: Desktop Browser to Extension WS (Same Machine)
Best for quick UI validation when running the PWA on the same machine as VS Code.

Checklist:
1. Enable the extension WS server:
   - Set `skillInstaller.ws.enabled: true` and `skillInstaller.ws.port: 0` in VS Code settings.
   - Confirm the port in the status bar (shows `WS:<port>`).
2. Start the mobile dev server:
   - `cd mobile-companion`
   - `npm run dev`
3. Configure Mobile to reach the WS server:
   - Set `VITE_RELAY_URL=ws://127.0.0.1:<port>` (use `.env.local` or shell env var).
4. Open the app at `http://localhost:5173` in the same machine’s browser.

Gaps to resolve:
- The extension requires a JWT token; the Mobile app currently has no pairing flow to retrieve one.
- Message format mismatch: Mobile wraps JSON-RPC in `{ type: "request", payload: ... }` while the extension expects JSON-RPC at the top level.

### Path B: Phone/Tablet to Extension WS (Same LAN)
Best for real device testing without relay.

Checklist:
1. Follow Path A steps 1-2.
2. Use the host machine LAN IP and firewall-allowed port:
   - `VITE_RELAY_URL=ws://<LAN-IP>:<port>`
3. Open the Mobile app from the device and ensure it can reach the host.

Gaps to resolve:
- Same as Path A (JWT pairing + message format mismatch).
- Mobile devices cannot reach `127.0.0.1` from the host machine.

### Path C: Mobile -> Cloud Relay -> Extension (Remote or Same Machine)
Best for production-like testing once relay is complete.

Checklist:
1. Start the relay service (local or hosted).
2. Point Mobile to relay:
   - `VITE_RELAY_URL=wss://<relay-host>` (or `ws://` for local).
3. Ensure the relay can route to the extension and translate messages.

Gaps to resolve:
- OAuth exchange endpoint `POST /auth/github/callback` is expected by Mobile but is not implemented in cloud-relay.
- Message format mismatch between Mobile and extension may require relay translation or a shared protocol.

## GitHub Login Feasibility
- The mobile app redirects to GitHub and expects a backend to exchange the OAuth code due to CORS constraints. See [mobile-companion/src/services/authService.ts](mobile-companion/src/services/authService.ts).
- The extension supports GitHub OAuth for VS Code login using `vscode://` callbacks; it is not a token exchange backend for the mobile app. See [vscode-skill-installer/docs/oauth-setup.md](vscode-skill-installer/docs/oauth-setup.md).

To make GitHub login work locally for Mobile Companion, you will need:
1. A token exchange service (relay or dedicated backend) that implements `/auth/github/callback`.
2. A GitHub OAuth App with callback URLs for:
   - `vscode://sofreshx.skill-installer/auth/callback`
   - `http://localhost:5173/auth/callback` (mobile dev)
   - `https://<relay-host>/auth/callback` (hosted relay)

## Audit Findings
- Protocol mismatch: Mobile wraps JSON-RPC in `{ type: "request", payload: ... }` but extension expects JSON-RPC at the top level.
- Auth mismatch: Mobile expects a relay token exchange; extension uses its own JWT tokens. There is no pairing flow to obtain extension JWTs from Mobile.
- Relay API mismatch: relay docs list OAuth endpoints but relay code does not currently implement them.
- Env var drift: build pipeline uses `VITE_WS_URL`, while Mobile uses `VITE_RELAY_URL`.
- Storage mismatch: security doc describes encrypted IndexedDB, but Mobile auth currently uses `localStorage`.

## Recommended Local Testing Strategy (Near-Term)
- Use desktop browser testing with the Mobile dev server for UI flows.
- For connectivity, implement a pairing/token issuance path in the extension, or add a relay adapter that translates Mobile messages into extension JSON-RPC.
- Use Playwright E2E for UI smoke checks (see [docs/e2e-setup-guide.md](docs/e2e-setup-guide.md)).

## Questions to Resolve Before Full Local E2E
- Should Mobile speak directly to the extension (same protocol), or should relay be the strict integration point?
- Where should OAuth token exchange live for Mobile in development: extension, relay, or a small local proxy?
- Do we want a QR pairing flow that carries both `ws://` URL and a short-lived JWT from the extension?
